# UC-03 — Enterprise fin-svcs maps SOC 2 → 800-53 Moderate

## Scenario

A B2B SaaS for commercial banks (FinPay Treasury Cloud) is preparing a federal agency pilot. The federal customer requires an 800-53 Moderate mapping. The SaaS has a SOC 2 Type II report and a PCI DSS v4.0 attestation. The mapping produces a crosswalk, identifies the gap controls, and produces a remediation plan.

The expected outcome:

- **71% overlap** between SOC 2 controls and 800-53 Moderate controls (the industry-typical range; allow ±3% for mapping variations).
- **94 gap controls** that must be added or remediated to meet 800-53 Moderate.
- **Remediation plan** with priority, owner, and target date for each gap.

## Walk-through

### Step 1 — Categorize (chunks/02-categorize.md §Procedure)

The SaaS processes financial PII and wires/ACH for ~1.5M accounts. Categorization:

- **C: MODERATE** — financial PII, wire details, account numbers.
- **I: MODERATE** — wire instructions must be integrity-protected.
- **A: MODERATE** — wire transfer windows are time-sensitive.

**System security category: MODERATE.**

### Step 2 — Baseline selection (Skill §4.2)

Baseline: **Moderate** (800-53 Rev 5, ~325 controls — derived; verify against SP 800-53B / FedRAMP baseline workbook).

### Step 3 — Crosswalk SOC 2 → 800-53 (chunks/09-crosswalk.md §Procedure)

The crosswalk in `data/crosswalks/soc2-to-800-53-mod.json` maps each SOC 2 Common Criterion (CC1–CC9) and Points of Focus to the corresponding 800-53 controls. The mapping is not 1:1; multiple SOC 2 criteria map to one 800-53 control and vice versa.

Summary of the crosswalk:

| Metric | Value |
|--------|-------|
| SOC 2 Common Criteria (CC) | 9 (CC1.1 – CC9.2) |
| SOC 2 Trust Services Criteria (incl. Points of Focus) | ~64 |
| NIST 800-53 Moderate controls (Rev 5) | 325 |
| Mapped controls (800-53 controls satisfied by some SOC 2 control) | 231 |
| Gap controls (800-53 controls in scope NOT satisfied by SOC 2) | 94 |
| Overlap % | 231 / 325 ≈ **71%** |

The overlap of 71% reflects the typical industry pattern. SOC 2 covers the governance, risk, and access-control core. 800-53 adds depth in:

- **Privacy controls** (PT family, in Rev 5) — not in SOC 2.
- **Supply chain risk management** (SR family, Rev 5) — not in SOC 2.
- **Program management** (PM family) — partial overlap.
- **Specific enhancements** (e.g., AC-2(11), AC-2(12), AC-2(13) on usage conditions, account monitoring, account disabling for high-risk users) — partial overlap.
- **Specific operational controls** (e.g., SI-4(2) system monitoring — automated tools, SI-4(7) detected events, IR-4(1) automated IR) — partial overlap.

### Step 4 — Gap list and remediation (Skill §6.2 + remediation plan)

The 94 gap controls are listed in `data/seeds/uc-03-gap-list.json` with each control tagged with:

- `control_id` (e.g., `AT-2`, `RA-3`, `SI-4`)
- `gap_type` (privacy / supply-chain / program-mgmt / specific-enhancement / operational)
- `priority` (High / Medium / Low)
- `owner` (role)
- `target_date`
- `remediation_action` (specific, testable)
- `evidence_refs`

A representative subset:

| Control | Gap type | Priority | Owner | Target | Remediation |
|---------|----------|----------|-------|--------|-------------|
| AT-2 | operational | Medium | HR / IT | 60d | Document quarterly security awareness training; track in LMS |
| AT-3 | operational | Medium | HR | 90d | Role-based training for ISSO, developer, admin; track in LMS |
| RA-3 | operational | High | CISO | 30d | Risk assessment per 800-30 Rev 1; update annually |
| SI-4 | operational | High | CISO | 60d | Deploy SIEM with continuous monitoring; document in SSP §13 |
| SI-4(2) | operational | High | CISO | 60d | Add automated threat detection |
| SR-1 | supply-chain | High | Procurement | 90d | Document SCRM policy |
| SR-3 | supply-chain | High | Procurement | 90d | Supply chain risk assessment |
| PT-7 | privacy (Rev 5 privacy) | Medium | Privacy Officer | 60d | Document PIA; agency approval |
| AC-2(11) | enhancement | Medium | IT | 60d | Account usage conditions documented in handbook |
| IR-4(1) | enhancement | High | CISO | 60d | Automated IR playbook |

### Step 5 — Shared-responsibility / inheritance (Skill §4.6)

The SaaS runs on AWS. The inheritance map is:

- **AWS-inherited** (commercial, not FedRAMP): physical, environmental, hypervisor, base IAM. The 800-53 mapping lists these as "inherited from cloud provider (AWS SOC 2 / ISO 27001 — to be cited)."
- **System-specific** (the SaaS implements): application-layer AC, AU, IA, SC, SI, IR.
- **Customer (agency)** (the agency implements): agency-side network controls, agency-side identity (federated via SAML/OIDC), agency-side audit log review.

The customer responsibility matrix (CRM) is a deliverable. The 800-53 mapping is the crosswalk; the CRM is the operating-model artifact.

## Variations / edge cases (test_adversarial.py)

- **V1 — PII volume increase:** if PII volume increases such that the categorization moves to HIGH, the gap list grows to ~140 controls. The skill must re-baseline and re-crosswalk.
- **V2 — PCI overlay changes the inheritance:** if the SaaS adds PCI DSS scope, the crosswalk changes (some SOC 2 controls are reinforced by PCI; some new gap controls appear). The skill must re-evaluate.
- **V3 — Customer (agency) wants FedRAMP:** the crosswalk may be insufficient; the agency may require the SaaS to be FedRAMP-authorized. The skill should produce a roadmap from SOC 2 → SOC 2 + 800-53 → FedRAMP.
- **V4 — Rev 5 privacy controls:** if the agency requires Rev 5, the gap list grows by ~20 privacy controls (the PT family). The skill must include Rev 5 overlays.
- **V5 — Sub-processor changes:** if a sub-processor (Auth0, Snowflake) changes its SOC 2 status, the inheritance map changes; controls that were inherited may become system-specific. The skill must re-evaluate.
